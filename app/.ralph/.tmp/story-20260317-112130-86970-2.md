### US-002: SQLite schema, migrations, and FTS5 triggers
Status: in_progress
Depends on: US-001

Description:
As a developer, I want SQLite tables and FTS triggers created automatically on startup so the app has persistent searchable storage without manual setup.

Acceptance Criteria:
- [ ] server/src/db/schema.js exports runMigrations(db)
- [ ] runMigrations creates the assets table using CREATE TABLE IF NOT EXISTS with all fields in the data model
- [ ] runMigrations creates assets_fts: CREATE VIRTUAL TABLE IF NOT EXISTS assets_fts USING fts5(id UNINDEXED, title, description, tags, location, creator, category, subcategory, filename, content='assets', content_rowid='rowid')
- [ ] runMigrations creates three triggers: assets_ai (AFTER INSERT), assets_au (AFTER UPDATE), assets_ad (AFTER DELETE) to keep assets_fts in sync
- [ ] server/src/db/index.js opens dam.sqlite at project root using better-sqlite3, calls runMigrations(db), exports db as default
- [ ] server/src/index.js imports db on startup (triggering migrations)
- [ ] Jest test: runMigrations on an in-memory DB (new Database(':memory:')) creates both tables without error
- [ ] Jest test: INSERT a row into assets → SELECT from assets_fts WHERE assets_fts MATCH the inserted title returns that row
- [ ] Jest test: UPDATE assets row title → FTS reflects new title
- [ ] Jest test: DELETE assets row → FTS no longer returns it
- [ ] Jest test: calling runMigrations twice does not throw
- [ ] Negative case: if dam.sqlite path is not writable, process logs the error and exits with code 1
